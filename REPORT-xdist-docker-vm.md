# Отчёт: pytest-xdist × ресурсы Docker VM × тюнинг Postgres (замеры для части 2)

Дата: **2026-06-13** (перемер на актуальном master). Стенд: MacBook Pro 14" **M4 Max, 14 ядер (10P+4E), 36 ГБ**, Docker Desktop 29.5.3.
Ветка: `feature/fast-tests-xdist`, актуализирована к master (merge `a6e4771c`). Сьют: **3477 тестов**. pytest 9.0.2, pytest-xdist 3.8.0.

> ⚠️ **Что изменилось против захода 12.06.** Тот прогон мерился с очисткой БД через `TRUNCATE ... RESTART IDENTITY CASCADE` — ошибка ветки: master уже перешёл на per-table `DELETE`. TRUNCATE искажал замеры (тяжёлый DDL с ACCESS EXCLUSIVE + работа с каталогом на каждом тесте). Ветка приведена к master (`DELETE`, commit `4d4fb1cc`), эксперимент перегнан целиком. **Это поменяло три вывода** — см. TL;DR. Старые цифры с TRUNCATE невалидны и заменены.

## TL;DR

> **Лучший результат: среднее/медиана 28 с — `-n 14 --dist load`, дефолтная Docker VM (~8 ГБ), дефолтный Postgres. Ускорение ×3.36 против честного последовательного прогона (94 с). Плато n8–n14 (28–29 с) — точное N не критично; `-n auto` (=14) попадает в оптимум без конфигурации.**

Три вердикта на чистых данных (3 плеча × 11 конфигов × 3 прогона, все 99 прогонов PASS 3477/3477, ноль флаков):

1. **16 ГиБ для Docker VM не дают ничего.** Дельты vm8 vs vm16 на всех конфигах ±1–2 с (±2%) — шум. Дефолтных ~8 ГБ достаточно; «накинуть памяти VM» — карго-культ для этого профиля нагрузки. (Без изменений против 12.06.)
2. **Тюнинг PG (fsync=off, synchronous_commit=off, full_page_writes=off, tmpfs) не помогает — и на малом параллелизме даже ВРЕДИТ.** На DELETE-очистке pgfast медленнее pgdef: serial −5% (94→99 с), n2 −11% (57→63 с), n4 −8%; на плато n8–n20 паритет (0%). **Это разворот вывода 12.06**, где pgfast «выигрывал». Причина: тот выигрыш был артефактом TRUNCATE — durability-тюнинг удешевлял именно тяжёлые DDL-коммиты CASCADE. С лёгким DELETE в одной транзакции тюнить нечего, а накладные tmpfs-контейнера дают лёгкий минус. **PG-тюнинг для этого сьюта — не «турбо-кнопка» (раскрывается в части 3).**
3. **Кривая масштабирования: плоское плато n8–n14 (28–29 с)**, колено на ~n8, мягкая деградация к n16/n20 (30–31 с). Оптимум — `-n 10`…`-n 14`.

**Разворот бонус-находки:** на DELETE-базисе **`-n 1` теперь МЕДЛЕННЕЕ `-n 0`** (97 vs 94 с, ×0.97) — появился небольшой положительный оверхед протокола execnet. На TRUNCATE-заходе serial был медленным (151 с) и master-процесс xdist его «обгонял»; с быстрым DELETE-serial (94 с) эта иллюзия исчезла — честный оверхед xdist на старте ~3 с.

## Методология

- **Изоляция** (код master): база-на-воркера (`DROP/CREATE DATABASE lms_gwN` + `create_all`), очистка между тестами — **`DELETE FROM` по всем таблицам в обратном FK-порядке, одна транзакция** (НЕ TRUNCATE — см. шапку), пул 5+5, Redis logical DB на воркера (`--databases 64`).
- **Postgres**: `postgres:latest` (18.x); дефолт + `max_connections=300` (плечи pgdef) либо + `fsync=off, synchronous_commit=off, full_page_writes=off, tmpfs` (плечо pgfast). Бюджет соединений: 20 воркеров × (5+5) = 200 < 300.
- **Свежие контейнеры PG+Redis перед КАЖДЫМ прогоном**, удаление через `docker rm -fv` + `docker volume prune -f` (см. инцидент №6 — без `-v` текут анонимные тома). Весь прогон 13.06: ровно 2–3 тома, дрейфа нет.
- **Кэпы памяти контейнерам**: PG 3g (4g при tmpfs), Redis 512m — голодающий контейнер умирает сам, а не валит OOM-киллером весь демон VM (инцидент №2).
- **Базовая линия — `-n 0`**, НЕ `-p no:xdist`: выгрузка плагина удаляет фикстуру `worker_id`, от которой зависят `db_config`/`redis_config`.
- `-p no:randomly --no-cov -q`; 3 повтора; медиана + min/max/среднее; в зачёте только PASS-строки.
- **Подавление сна**: весь прогон под `caffeinate -i`. Критерий валидности строки: PASS, passed=3477, wall−pytest < 15 с (разрыв wall/monotonic = маркер сна macOS).
- Демон Docker не рестартуется в happy path; только при реальной смене размера VM (1 раз — переход на vm16).

## Попарные сравнения (медианы wall)

### vm8 vs vm16 — память Docker VM: эффекта НЕТ

Все дельты в диапазоне −7…+3% без знака-тенденции (serial +2%, n4 0%, n8 0%, n10 0%, n14 −7%, n20 +3%). Профиль нагрузки не упирается в память VM: суммарный RSS PG+Redis < 1 ГБ, остальное — page cache, которого хватает и в 8 ГБ. **Рекомендация: дефолтную VM не трогать.**

### pgdef vs pgfast — тюнинг PG: не помогает, на малом N вредит

| Конфиг | pgdef | pgfast | дельта |
|---|---:|---:|---:|
| serial | 94 | 99 | **−5%** |
| n1 | 97 | 104 | −7% |
| n2 | 57 | 63 | **−11%** |
| n4 | 38 | 41 | −8% |
| n6 | 32 | 34 | −6% |
| n8 | 29 | 30 | −3% |
| n10–n12 | 29 | 30 | −3% |
| n14 | 28 | 28 | 0% |
| n16–n20 | 30/31 | 30/31 | 0% |

На DELETE-очистке pgfast НЕ выигрывает ни одного конфига, а на serial–n6 проигрывает 3–11%. Вывод 12.06 («pgfast быстрее, serial −21%») был артефактом TRUNCATE: durability-тюнинг + tmpfs удешевляли тяжёлые `TRUNCATE...CASCADE`-коммиты, которых на каждый из 3477 тестов было по одному. Убрали TRUNCATE → удешевлять нечего, а накладные tmpfs-контейнера дают лёгкий минус на малом параллелизме; на плато всё тонет в дележе CPU. **Гипотеза пользователя «без тюнинга вроде быстрее» теперь ПОДТВЕРЖДЕНА на корректной очистке.** Тюнинг PG целиком уходит в часть 3.

### Почему лучшая точка именно такая

- **28 с на n14 (оба vm8-плеча)**: 10 P-ядер насыщены, +4 воркера утилизируют E-ядра в окнах I/O-ожидания. Плато n8–n14 (29/29/29/28) шириной в 4 точки — точное N некритично.
- Деградация за плато мягкая (n20 = 31 с): сьют достаточно I/O-bound, чтобы oversubscription не обваливал время, но выигрыша уже нет — гипотеза «выиграем за пределами ядер» не подтвердилась.
- **Потолок ускорения ×3.36 при 10 P-ядрах** — далеко от идеала ×14. Доминируют входной налог воркеров (session-setup × N) + контеншн на общих ресурсах, не голый CPU.
- **Прагматичная рекомендация читателю: `-n auto` (=14) на дефолтном Docker и дефолтном PG — 28–30 с, в оптимуме, ноль конфигурации.**

## A5½. Микробенчмарк очистки БД между тестами (ответ на коммент Tishka17)

Отдельный замер, отвечающий на вопрос из комментариев части 1: какой способ «обнулять» БД между тестами дешевле. Сравнивались **три стратегии per-test**, на реальной схеме проекта (53 таблицы), zero rows (реальное per-test состояние), медиана за цикл (скрипт `cleanup_bench.py` + `run_cleanup_bench.sh`):

| Стратегия | 40 циклов, мед. | 300 циклов, мед. | max (300) | vs DELETE |
|---|---:|---:|---:|---:|
| **DELETE** по таблицам, 1 транзакция | 9.33 мс | **9.24 мс** | 11.9 (стабильно) | ×1.00 |
| **TRUNCATE** … RESTART IDENTITY CASCADE | 12.36 мс | 12.58 мс | **34.7** (хвост скачет) | ×1.36 |
| **TEMPLATE** — честный per-test `DROP DATABASE` + `CREATE … TEMPLATE` | 31.06 мс | **33.23 мс** | 67.1 | **×3.60** |

Выводы:
1. **DELETE — самый дешёвый и самый стабильный** (max 11.9 мс, плотно). Подтверждает решение части 1.
2. **TRUNCATE +36% к медиане и нестабильный хвост** (max 34.7 мс = ×2.7 медианы) — ровно та «нестабильность TRUNCATE» из части 1, теперь изолированно: DDL с ACCESS EXCLUSIVE + работа с каталогом, стоимость зависит от его состояния.
3. **Честная per-test TEMPLATE-изоляция стоит ×3.6 (+24 мс/цикл)** — потому что `CREATE DATABASE … TEMPLATE` требует НУЛЯ подключений к БД, а session-scope пул держит их постоянно. Значит на каждый тест надо гасить пул и реконнектиться — это возврат того самого «входного налога», который мы убрали в части 1. Замер подтверждает интуицию из ответа Tishka17 цифрой.

На масштабе сьюта (3477 тестов, только очистка): DELETE ≈ 32 с, TRUNCATE ≈ 44 с (+12 с), TEMPLATE ≈ 116 с (+84 с). Для xdist это и есть аргумент в пользу `DELETE` + базы-на-воркера (создаётся один раз на воркер), а не пересоздания БД на каждый тест.

## Журнал инцидентов (хронология построения харнесса, заходы 11–12.06)

> Числа в этой и следующей секциях — из ранних заходов с TRUNCATE/грязным диском; как иллюстрация методологии они валидны (механика инцидентов от очистки не зависит), но это НЕ опорные цифры статьи (опорные — в таблицах выше, перемер 13.06 на DELETE).

1. **Idle-sleep ноутбука**: без caffeinate macOS уснул, wall≫pytest (monotonic во сне стоит), числа — мусор. Фикс: `caffeinate` + критерий wall−pytest<15 с.
2. **OOM-смерть демона Docker VM**: ~17 циклов пересоздания контейнеров без кэпов памяти → VM OOM-kill демона (`Exited 137`), харнесс записывал мусор (connection refused). Фикс: `--memory`-кэпы + реальные probes + ретраи + громкий ABORT.
3. **Гонка осиротевшего прогона**: kill драйвера не убил дочерний `xdist_arm.sh`; два прогона делили порты 5434/6381 → ложные FAIL. Фикс: `stop_experiment.sh` (убивает дерево с листьев) + preflight.
4. **Тихая смерть после обновления Docker 29.5.3** + неуправляемый драйвер: упавшее плечо не остановило старый драйвер, он молча переключил VM на 16 ГиБ. Фикс: `|| exit 3` на каждом шаге, `set -x`-трасса в `.debug`, trap EXIT, минимум рестартов демона.
5. **bash 3.2 на macOS**: нет `exec {fd}`/`BASH_XTRACEFD`, пустой массив под `set -u` падает. Фикс: stderr→debug-файл, guard `${arr[@]+...}`.
6. **Утечка анонимных томов postgres:18** — главный инцидент: образ объявляет `VOLUME /var/lib/postgresql`, `docker rm -f` БЕЗ `-v` оставляет том; ~70 прогонов × ~2 ГБ = **983 тома / 142 ГБ**, диск VM кончился. Фикс: `docker rm -fv` + `docker volume prune -f` на каждое пересоздание (после фикса: 1–3 тома на весь прогон).
7. **Гочка PGDATA в postgres:18**: дефолтный PGDATA переехал в `/var/lib/postgresql/18/docker`, поэтому `tmpfs: /var/lib/postgresql/data` из compose **молча не накрывал данные**. Фикс: явный `PGDATA=/var/lib/postgresql/data`.
8. **TRUNCATE вместо DELETE в коде ветки** (обнаружено 13.06): ветка `feature/fast-tests-xdist` несла `TRUNCATE...CASCADE`-очистку, тогда как master уже перешёл на `DELETE`. Искажало все замеры 12.06. Фикс: ветка приведена к master (`DELETE`, `4d4fb1cc`), эксперимент перегнан — текущий отчёт.
